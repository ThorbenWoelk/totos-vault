## Cryptography
"Server-minted one-time nonce"
- nonce is a number used once
- minted by server = Server creates random secret, not client. Client may know public handoff_id, but only server issues secret.

Creator binding: even if handoff_id leaks, wrong caller cannot redeem.

Redeem-once endpoint: Do not poll reusable status that returns creds many times.  
Instead: 
- browser completes Google login
- browser sends tokens once to backend
- Android app calls POST /api/auth/mobile-handoff/{id}/redeem
- backend checks creator binding + not expired + not redeemed
- backend returns success payload once, then deletes handoff  

## Predicates

**Rust / cfg:** `#[cfg(predicate)]` controls conditional compilation.

- If the predicate is `true`, the annotated code exists.
- If the predicate is `false`, the code is removed before compilation.
- `predicate` here simply means the condition being checked.

Common predicates:
- `target_os = "linux"`
- `feature = "serde"`
- `test`
- `debug_assertions`
- `unix`

## State bags vs. and strict controller APIs

**Frontend architecture / strict controller API:** In TypeScript and Svelte apps, let routes and components read controller-managed state, but route all state changes through clearly named controller actions.

- UI reads state, controllers own mutations.
- This centralizes behavior and helps preserve invariants.
- It also makes larger refactors safer, because behavior changes happen in one controlled place instead of being spread across components.

## Frontend state management

**Frontend state management / React vs Redux vs Svelte:** React builds the UI; Redux manages application state.

- They are often used together, but Redux is not React-specific in principle.
- In Svelte apps, Redux is much less common because Svelte already provides simpler built-in reactivity and stores.
- In Svelte, a lot of state can just live in reactive state like `let count = $state(0);` or in stores like `const count = writable(0);`.

## Firestore cost optimization

**Firestore cost optimization:** Working the efficiency frontier means balancing reads, indexes, caching, and data size.

- Firestore pricing is heavily driven by read and write operations, so every query and index decision has a direct cost impact.
- Optimizing means choosing the right query pattern, keeping indexes lean, using caching where it makes sense, and structuring data to minimize the number of reads needed.
- The core idea: the cheapest Firestore query is the one you never make.

## Instrumentation

= modifying code so it reports on itself while it runs.

You take the original code and inject extra statements that don't change behavior, only record what happened. Like putting sensors on a machine.

2 Types:
- **Istanbul** = source-level instrumentation. A build plugin rewrites your JS at compile time. Accurate, slow, bloats the bundle. Use only in a test build.
- **V8 coverage** = engine-level. Chromium's JS engine already tracks execution internally; you ask it for the data via Chrome DevTools Protocol. No code rewriting, less accurate at the branch level, needs source maps to translate bundled positions back to your source files.

Instrumentation changes your bundle. Source maps can drift, async code gets odd counters, dead-code elimination fights the instrumentation, and the numbers mean "executed" not "tested" - a page load hits thousands of lines without asserting anything about them.
