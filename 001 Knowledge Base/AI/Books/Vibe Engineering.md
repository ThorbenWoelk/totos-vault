## 1.4 Defining a new discipline: Vibe Engineering

The answer to the chaos of alchemy and the dangers of vibe coding is _vibe engineering_. If vibe coding is alchemy, then vibe engineering is applied chemistry - systematic, rule-based, and grounded in measurable evidence. It’s a discipline that combines the creative power of LLM prototyping with the non-negotiable pillars of professional software engineering: rigorous testing, quality assurance, solid error handling, attention to edge cases, fulfillment of non-functional requirements (performance, scalability, security), and production stability.

This concept is an evolution - not a rejection - of timeless engineering wisdom. As Martin Fowler once put it: “Any fool can write code that a computer can understand. Good programmers write code that humans can understand.” In the AI era, this statement takes on new weight: we must create not just code, but entire systems that are understandable, verifiable, and safe - even when parts of them are generated by a machine.

That’s maybe the biggest irony: this entire AI-driven engineering wave has simply reminded us that AI thrives in the same conditions human developers do - systems with clear boundaries, crisp contexts, and a reliable chain of verification. This perspective emphasizes that AI is a powerful tool in the hands of the engineer, not a replacement for them.

### 1.4.1 Two faces of the vibe: coding vs engineering

When the rapid experimentation becomes too costly and the illusion of speed ends in disaster, a practical question arises: how do we tame this new, powerful force? The answer starts with a clear distinction between two approaches that are often lumped together: _vibe coding_ and _vibe engineering_. This is more than a semantic quibble - it’s the difference between exploration and construction, between a sketch and an architectural blueprint, and choosing between them comes with a set trade-offs. Both are useful, but confusing the two is a mistake, one of the main causes of the failures we described earlier.

#### Vibe coding: the creative flow

Vibe coding is a dance with uncertainty. It’s a prompt-driven, intuition-first mode in which the developer surrenders to iterative discovery to capture a desired result - the vibe.

As Andrej Karpathy put it, it’s the act of “leaning into the vibes, accepting what the AI gives you without checking every line, and if you hit a bug you paste it back into the model - eventually it works.” In that world, intention matters more than implementation detail and the code becomes almost secondary to the dialogue with the machine.

Vibe coding shouldn’t be demonized - in the right context, it’s invaluable: the digital sketchbook for rapid prototyping, a fast path to an MVP, a way to explore UI ideas, or a tool for producing repetitive boilerplate. It turns a foggy idea into a working interface in minutes rather than days. Trading off speed for the technical debt - essential at the beginning of a project when you’re discovering what you actually want to build. It’s a form of triage - if you do not know if the particular feature/idea survives in the long term, losing time on solidifying the implementation is a clear mistake.

However, the very strengths that make vibe coding powerful for discovery become its greatest liabilities for production systems. This approach often generates code that is brittle, opaque, and laden with technical debt. Since the developer may not fully understand the implementation choices made by the AI, maintaining, debugging, and extending the codebase becomes a significant challenge. Furthermore, AI-generated solutions are optimized for functional correctness, not performance, security or maintainability. This can lead to inefficient algorithms, unnecessary and complex abstraction layers, poor resource management, and hidden vulnerabilities that are unacceptable in a production environment. Consequently, while vibe coding is an excellent tool for scaffolding, that scaffold must be dismantled and replaced with deliberately engineered, robust, and well-understood code before a project can be considered production-ready.

#### Vibe engineering: disciplined practice

If vibe coding is the sketch, vibe engineering is the architectural plan. It’s the deliberate, systematic integration of AI into a mature software lifecycle.

This principle is rooted in the fundamental nature of the technology itself. Whether the tool is Gemini, ChatGPT, or Claude, all current-generation LLMs share the same core architecture: they are probabilistic, non-deterministic systems. This shared, inherent weakness is precisely why a provider-agnostic _methodology_ is not just beneficial, but essential. The problem to be solved is not a specific model's flaw that will be patched in the next release but the characteristic of the entire technology class. Crucially, these systems are statistical: they predict likely next tokens, not objective facts, so they do not contain a built-in _model of truth_. That absence forces engineering trade-offs - for example, increasing model size or _temperature_ (creativity of the model) may improve fluency but raise hallucination risk. On the other hand, adding retrieval improves factuality but increases complexity and surface area for failure. In short, every design decision trades one axis (accuracy, creativity, safety, cost, latency) against another.

The discipline of vibe engineering is designed to address this reality directly. It provides a framework to _wrap_ this probabilistic core with a deterministic shell of human intent. The executable specification becomes the rules for agents, and a verifier (e.g. the CI/CD pipeline) becomes the impartial enforcer of those rules. Within this structure, the specific LLM used to translate the specification into code becomes a secondary concern - a choice based on cost, speed, or stylistic preference, not a fundamental determinant of correctness or safety. The shift from alchemy to science is a shift from ingredient-obsession to process-mastery. That’s why “science” of vibe engineering rests on concrete, repeatable, and measurable techniques that form the foundation of this book. These include:

_Systematic prompt engineering:_ Provide an LLM with code slices and tests, so the model can return an endpoint/UI that compiles and passes immediately

_Retrieval-augmented generation and grounded retrieval:_ A support assistant answers from product docs and release notes it actually cites, cutting hallucinations and surfacing anchors.

_PR review checklist:_ Let the model do a first-pass review using a fixed checklist (input validation, auth, obvious performance issues). It leaves comments; humans approve and own the change.

_Incident triage:_ When an error hits, the model summarizes logs, points to the likely file/line, and drafts a tiny, reversible fix with clear rollback steps in the PR.

_Guarded automation:_ An agent opens a dependency-update PR with a change summary and test plan. It merges only after required approvals/CI checks and auto-rolls back if health metrics drop.

Introducing these techniques fundamentally reshapes the role of the developer - from _code author_ to _system designer and validator_. The core action is no longer writing individual lines of code, but designing, testing, and safeguarding a complex human-AI system that produces the final artifact - software with proper functional and non-functional requirements. The value of an experienced engineer shifts up one level of abstraction: no longer in manually crafting each _part_ (code), but in designing the _factory_ (a continuous, safe, AI-powered workflow) that produces those parts.

However, with solving one set of challenges, here comes the new ones. Even in the best-run teams and projects, AI doesn’t just accelerate familiar work - it also introduces a new class of risk we didn’t face before. As automation scales, human attentiveness tends to erode, and the burden quietly shifts from “doing more” to “proving it’s still safe to do more.” Once the core job is orchestrating a human-AI system, the question is no longer “Can it ship?” but “Can we trust what ships?”. Because when that vigilance slips, even slightly, the system accrues a different kind of liabilities.

This flaw is _automation bias_: our tendency to over-trust and uncritically accept information from automated systems, even when our own expertise might suggest otherwise. It's the silent assumption that the machine is right, which turns a helpful assistant into a potential single point of failure. This growing liability is a new kind of debt we are only now beginning to understand.

### 1.4.2 Trust: a new kind of debt

Every engineer knows the concept of technical debt - the hidden cost we incur tomorrow for shortcuts we take today. In the age of AI, a subtler twin has emerged: _trust debt_. It’s the accumulated, invisible cost of shipping AI-generated code without adequate verification.

Trust debt grows quietly. The initial productivity win is visible and celebrated - new functionality delivered in a day instead of a week. The cost - weeks of a senior engineer’s time spent debugging a performance anomaly, patching a security gap, or refactoring code that violates architectural principles - stays hidden and arrives later.

This debt is a direct byproduct of the _dump-and-review_ culture. This approach uses AI to generate a large slab of code, opens a pull request, and implicitly offloads responsibility for verification to the reviewer. It’s classic diffusion of responsibility: the presence of the AI (“the model wrote it”) and a reviewer (“someone will check it”) dilutes the author’s ownership of quality.

It’s the same structural failure we’ve seen with autonomous vehicles - less spectacular in its consequences, but fundamentally similar. There’s also a well-documented human factor at play: when oversight is reduced to rare, high-stakes interventions, vigilance erodes. Psychologists call this _vigilance decrement_; Human-Machine Interface researchers talk about _automation complacency_ and the _out-of-the-loop problem_. The mind relaxes when nothing seems to require it, attention drifts, and the signal-to-noise threshold rises. By the time a true anomaly appears, reaction times are slower, working memory lacks the necessary context, and the reviewer must rebuild a mental model under pressure - precisely when mistakes are most costly.

In code review, that looks like scanning machine-generated diffs on autopilot, over-trusting green tests, and missing the one invariant violation that matters. Late handoffs amplify this: the human is asked to jump from near-zero involvement to deep scrutiny at the worst possible moment. Without mechanisms that keep humans engaged and informed throughout, vigilance doesn’t just fade - it switches off. A reviewer faced with a monolithic, agent-authored PR is the cognitive twin of a safety driver during an unexpected disengagement. Both are forced into real-time sense-making of behavior they didn’t author, with limited visibility into the system’s internal state. The result is the same: delayed reaction, shallow scrutiny, and a higher chance of missing rare-but-critical failure modes. Just as mileage-driven “it usually works” is not a safety case for cars, “the tests happened to pass” is not a safety case for code.

That’s why trust debt is an economic externality of AI-assisted development. The individual who leans on it enjoys immediate benefits - speed and a closed ticket - while the hidden trade-off is that the costs are socialized across the team, disproportionately taxing the most experienced (and expensive) engineers. They become the last line of defense, paying down the debt in overtime and during stressful production incidents. What feels productive at the individual level becomes value-destructive at the organizational level.

The discipline of vibe engineering must therefore be aimed at eliminating this tension entirely. We need to resolve the trade-off by mandating a strict replacement of "dump-and-review" with "verify-then-merge" treating all prompts and outputs as fully accountable artifacts, enforcing executable specs as non-negotiable contracts. The goal isn’t “no debt” (that’s impossible) but a deliberate, auditable workflow that keeps trust debt small, visible, and payable - and turns inevitable errors into fast, low-risk feedback.

To make that concrete, take a simple task: write a Python function that validates a 13-digit ISBN. It must accept strings with or without hyphens, compute the checksum per the official algorithm, and return a boolean.

Now, imagine the typical approach: we start by writing a prompt to generate the function.

Generate a Python function is_valid_isbn13(isbn_string) that validates a 13-digit ISBN based on the official checksum algorithm. The function must correctly handle input strings that may contain hyphens. Ensure the function returns True for valid ISBNs and False otherwise.

Sounds legit, isn’t it? The model might produce something that seems to work, but it will likely fail on various edge cases. Because we are dealing with a probabilistic model, running the same prompt again will produce a _different_ implementation, often with an entirely new set of bugs and failure modes. Let’s generate this function with the GPT-5 model.

### 1.4.4 Stuck in old ritual: what stalls real adoption

However, the popular, fundamental error is assuming AI-assisted coding adoption in production systems is just a technology problem. Yes - today’s models are gaining capabilities slower, but they are already powerful, and the tools keep improving. Meanwhile, the real bottleneck is often human and organizational: new capability colliding with old habits, legacy processes, and a need for control. Treating AI pilots as tech rollouts rather than change-management efforts is why so many stall: they optimize models while leaving incentives, workflows, and accountability untouched.

Cultural resistance and fear of change form the first wall. Introducing AI raises anxieties about job loss and reluctance to trust systems whose inner workings aren’t fully understood. Engineers, proud of their craft, can read AI as a threat to their autonomy and expertise. Just as strong is process inertia. Development teams run on deeply rooted rituals - sprint planning, daily stand-ups, code reviews. AI-driven workflows disrupt that cadence, forcing a redesign of how work is defined, assigned, and verified, while integration with existing (often aging) CI/CD adds technical friction. And with that, here comes the awkward questions nobody wants to answer.

For example, if our “velocity” flips halfway through a sprint because an agent just rewrote a subsystem overnight, what velocity is the agent running at? Does it sprint on Fibonacci points or on GPU hours? What if it is running in a continuous way and doesn’t “commit to the sprint”, cheerfully ignores your planning session, operates across weekends, and drops a 3,000-line PR at 02:17. Your burndown chart starts to look less like progress and more like an ECG… and probably much healthier than one from your company's Agile Coaches and Project Managers.

What does a retro even look like with an always-on contributor that has no feelings, no context switch tax, and no coffee breaks? You can’t ask it how it “felt” about the sprint. Instead, you’re left reviewing artifacts: prompts, traces, test diffs, policy-gate outcomes, and rollback drills. “What went well?” needs to become “Which guardrails caught what?” “What didn’t go well?” becomes “Where did we trust too much and verify too little?”

The punchline is that for many organisations, the list of second-order problems is longer than the first-order wins. Our muscle memory kicks in and we respond with familiar rituals, and you end up adopting the tech at the shallowest layer: a thin layer of AI on top, the same old process underneath, and a growing pile of “mysterious” productivity losses that aren’t mysterious at all.

There is also another, an engineering part of that equation - a fundamental need for control and reproducibility. Software engineering, at its core, seeks deterministic, predictable systems, while language models are inherently probabilistic. Without solid specifications and tests, AI output is a “black box,” which is unacceptable to engineers who must understand why a system behaves the way it does.

In this context, new practices become the human bridge over the cognitive gap between human intent and probabilistic machine output. It’s the breakdown of our traditional rituals of trust. In the classic model, trust grows through interaction and shared context. A senior engineer reviewing a junior’s code can ask, “What did you mean here?” - a conversation that reconstructs the author’s mental model and builds trust. You can’t ask the same question of an LLM, at least not the old way - its output isn’t the product of reasoning but of statistical matching. That breaks the old trust loop.

So some organizations, unsure how to trust the new tool, retreat to what they can control: they limit usage or ban it outright over security and privacy concerns. That response is a symptom of mistrust. Effective adoption therefore demands new rituals - rituals that rebuild trust on a new foundation. We will tackle this topic directly in the book, presenting a step-by-step guide on how to ensure the successful adoption of these tools within your own application and team.

## 1.5 A new mental model for vibe engineering

So how do you avoid trust debt and the chaos of unstructured vibe coding in practice? By adopting a disciplined methodology whose center of gravity is development led by an executable specification.

Even with all the mentioned contrast between vibe coding and vibe engineering, we still think Vibe Coding is useful as an exploratory tool. Nowadays, it should be considered a starting point for any more complex features, as there is no better way to understand the domain, then spiking, scoping and prototyping. The challenge is balancing this initial creativity with the discipline of production engineering. So how do you harness the creative chaos of vibe coding without accumulating trust debt? By adopting a methodology that formalizes the handoff from initial idea to owned code.

### 1.5.1 Practical example of using a cycle

There’s no better way to truly understand a concept than to walk through it step by step. So let’s look at a practical example of using the cycle on a real feature: “30-day refunds.”

The previously mentioned Vibe Coding part is crucial - it's where we gather the necessary information and insights to better plan the final feature. Do you remember our little trick of lacking a test-case for ISBN13 number validation? Any good prototype should be simple and ultimately thrown away, but used to give us the real rules, edge cases, data shapes, and the checks we’ll use to prove that our code works.

Armed with real experience (even if just “vibed”), we can begin specification. Before anyone opens an IDE, the product note “refunds within 30 days” should be turned into an executable behavior that nails down edge cases, money arithmetic, and reasons for rejection. The tools will vary by project - the important thing is that the pipeline can run them. For example, we can use a Gherkin specification (a plain-English Given–When–Then format, that can be used).

Feature: Refund policy

Rule: Time-bounded, single-order refunds

Example: Within 30 days is allowed

When a refund of 25.00 USD is requested at 2025-08-28T09:59:00Z

With intent pinned and verification gates already created, we can state additional constraints, more application specific ones - Kotlin/JDK 21, Spring Boot 3, Postgres, a hexagonal design, money represented as amountCents + currency, idempotency strategy. Rather than freehanding a design doc, we ask the agent to propose a concrete blueprint we can edit. Then we can work together with the agent, to revise the plan until we think we are ready. We can then ask our LLM to decompose the work into small, independent tickets the pipeline can judge without arguing about taste. We let the agent propose those tickets, each with a clear definition of done tied to the spec.

Split the approved blueprint into ≤ 2h tasks. For each task, output the goal, inputs (spec fragments), expected outputs (code/tests), and a Done checklist that CI can verify. Avoid cross-task coupling.

And now, finally, comes the Implement (Verify) phase - but the order matters: tests and checks first, then code that satisfies them.

For the policy task, the agent first writes tests (or uses existing ones) and implements the pure RefundPolicy, until the whole suite goes green. The agent generates the controller, DTO validation, and RestAssured tests that fail if any response deviates.

LLM can even create its own verification gates.

Publish a Markdown report to the PR with histograms and pass/fail verdicts.

What we’re doing here is a reincarnation of good old TDD - only zoomed out and made explicit for the AI era. Like TDD, we define the effect first and let implementation follow: intent before code, contract before commit. But it’s not just “tests first.” In Vibe Engineering the primary artifact shouldn’t be an unit test, but an executable specification that spans boundaries TDD never touched: API schemas and error semantics, domain invariants and property checks, state-machine guards, SLOs as perf gates, security and compliance policies, and provenance.

The human’s job shifts from hand-crafting lines to curating intent - writing the spec, approving the plan, and auditing results - while agents translate that intent into code and verification. _Red_ _→_ _Green_ _→_ _Refactor_ becomes _Specify_ _→_ _Implement_ _→_ _Verify_, where red can mean a failing contract test, a violated invariant, a mutation score below threshold, or a p95 latency gate tripping in CI - not just a broken unit.

We hear the objection: turning intuition into specs, tests, and contracts sounds exhausting. It would be - if we tried to do it all by hand. The twist is that the very AI pushing us toward discipline is also what helps generate, evolve, and enforce those artifacts. Which brings us to the practical question: what does a real-world toolchain look like when specifications are first-class?

### 1.5.2 Tools as force multipliers: IDE + CI/CD

We’re not hand-writing everything from scratch each time. In a healthy project, core concepts don’t churn weekly, stacks don’t reinvent themselves every sprint, and practices don’t reset to zero. Much of our work—spec patterns, review checklists, policy gates, even “guide-chains” of prompts—stays general and reusable. That’s why we lean on tools: to capture once, reuse often, and automate the boring-but-crucial parts. Modern IDEs are the cockpit for this: they don’t just chat with an LLM, they apply templates, run our guide-chains, stamp out scaffolds from spec kits, and dispatch agent tasks while streaming real-time feedback. The CI/CD pipeline is the factory floor that enforces the contract: spec-derived tests run on every commit, security and compliance packs interrogate artifacts, and policy gates apply uniformly across services - no exceptions, no heroics.

There is, however, a familiar trap: _the machine verifying the machine._ If the AI generates both code and tests, you can end up with a closed loop that blissfully misses the same edge case. The antidote isn’t more manual line-reading - it’s human attention at the _right level_: authoring and curating high-quality specifications, designing adversarial and mutation-resistant test plans, and maintaining shared, company-wide libraries of prompts, specs, and policies. In that sense creating specification isn’t busywork; it’s cognitive offloading, and fascinating one. We codify intent once, reuse it many times, let the tooling handle the mechanically repeatable work, and keep humans focused on what they’re uniquely good at - defining goals, context, and acceptable risk. Teaching you how to properly handle the verification of this code is another key goal of this book.

In our opinion, the upside is that this isn’t bureaucratic overhead - it gets genuinely fun once you learn the moves. As the team leans into spec-first habits, the frustrating mistakes fade: fewer late surprises, fewer “mystery” regressions, fewer review stalemates. The cognitive load shifts to where it belongs, and the work feels like craft again - clear intent, clean contracts, tight feedback loops, and the quiet satisfaction of a pipeline going green for the right reasons.

### 1.5.3 The winning loop - and the risks ahead

Even the most mature vibe engineering practice won’t eliminate risk, however with proper techniques we are able to change its shape. Because the most insidious danger is erosion of the shared mental model: if no one understands _why_ the system works, diagnosing critical incidents becomes guesswork. And under a tight deadline, there is always the gravitational pull back to dump-and-review mentioned before - the temptation to offload responsibility unless leadership continuously reinforces boundaries and practices.

That’s why finally, there always must come the _Own_ phase. The final code needs to be understood, documented, and owned by the engineering team. It is no longer “AI code”, it is our code - breaking the diffusion of responsibility, bringing accountability. The surface you verify is larger, the feedback is multi-dimensional, and the failure modes are more subtle (non-determinism, prompt drift, closed-loop testing). The ultimate challenge of Vibe Engineering isn’t technical so much as epistemological: managing the known unknowns (captured in the spec) and building resilience to the unknown unknowns (emergent from the AI’s black-box behavior). The refactor/own stages are your front line against the latter - refactoring forces engineers to deconstruct and reconstruct AI logic, often exposing hidden flaws.

However, what sounds easy in theory is really hard in practice. Ownership demands a deep, durable mental model so the team can reason about behavior when the unexpected happens.

## 1.6 Owning - The last mile of vibe engineering

Even with the most refined methodologies and powerful tools, the ultimate success of AI-assisted development hinges on confronting two fundamental, human-centric challenges. The first is the qualitative gap between rapidly generated code and a truly production-ready system. The second is the immense cognitive burden required to establish genuine ownership over code that a human did not author. These challenges represent the "last mile" of development, where the illusion of speed collides with the non-negotiable demands of engineering discipline.

### 1.6.1 The not-the-end-yet 70% Problem

A foundational principle for understanding the practical limits of current AI coding assistants is what Addy Osmani from Google has termed the 70% Problem. The concept posits that AI excels at accelerating the initial, well-trodden path of development - the first 70% - which primarily involves scaffolding features, generating boilerplate, and implementing common patterns. However, these tools falter when faced with the final 30%, which constitutes the most critical, context-sensitive, and high-stakes portion of engineering work. This dynamic creates a dangerous illusion of competence by making the easy parts of the job feel effortless, which can mask underlying knowledge gaps and create a false sense of progress toward a production-ready artifact. The 70/30 split is not merely a quantitative measure of work but represents a qualitative shift from tasks of generation, at which models excel, to tasks of _judgment_, which remain a uniquely human capability.

The "hard 30%" is not a monolithic block of work but a collection of distinct engineering disciplines that require deep contextual understanding and consequential reasoning. This is where the statistical pattern-matching of a language model proves insufficient.

_Edge Case Cartography:_ AI-generated code is exceptionally good at implementing the "happy path" but consistently fails to account for the edge cases, error conditions, and boundary scenarios that define robust, real-world applications. Production readiness is defined not by how a system works under ideal conditions, but by how it behaves at its boundaries when faced with rare inputs or unexpected user actions. This requires an understanding of what can go wrong, that current LLMs lack, as demonstrated by catastrophic data loss and security failures in the beginning of this chapter.

_Architectural Integrity and Integration:_ While individual AI-generated components may be locally correct, they often fail to integrate seamlessly into a larger system architecture. The generated code can violate project-specific conventions, introduce non-optimal solutions, or create a "less consistent architecture" that drifts from established patterns. This frequently leads to an "integration nightmare," where each generated component is described as "a perfect puzzle piece for the wrong puzzle". This failure to align with global system design creates a new, insidious form of technical debt. Unlike classic technical debt, which is often a conscious trade-off, this "implicit architectural debt" is silent and systemic. The code appears correct and may even pass unit tests, but its poor integration characteristics create liabilities in the seams between components, which only become apparent during a major refactoring or a stressful production incident. This is a technical manifestation of the "trust debt" discussed earlier.

_Comprehensive Verification:_ An AI assistant might generate basic unit tests that confirm the happy path, but the final 30% requires a much deeper level of verification. This includes writing property-based tests to probe system invariants, performance tests to validate service-level objectives (SLOs), mutation tests to ensure test suite quality, and security tests based on a threat model. These tasks demand a level of intent and adversarial thinking - actively trying to break the system in creative ways - that AI currently cannot replicate, while being useful in finding blind spots in our own thinking.

_The Security and Compliance:_ Security is one of the most glaring weaknesses in the final 30%. Research indicates that a high percentage of AI-generated code contains vulnerable patterns. Models learn from vast public code repositories, many of which contain insecure or outdated practices, and they reproduce these flaws without any understanding of their security implications. This leads to textbook vulnerabilities like SQL injection, hardcoded secrets, insufficient input validation, and weak authentication flows. Furthermore, AI models are blind to regulatory and compliance frameworks, such as those in finance or healthcare, making their unverified output unacceptable for use in systems where these rules are non-negotiable.

_Performance and Scalability:_ AI-generated code typically prioritizes functional correctness over performance and efficiency. The output can be non-optimal, containing inefficient database queries, poor algorithmic complexity, or excessive resource usage that does not scale under production load. This directly conflicts with the vibe engineering principle of treating performance not as an afterthought but as an executable contract, specified and verified as an SLO gate in the CI/CD pipeline.

This problem is also self-compounding and poses a systemic risk to the software ecosystem. At a micro level, the probabilistic nature of LLMs means that small errors in a long generation sequence cascade and compound, drastically reducing the chance of a correct outcome in complex tasks. At a macro level, as more of this "70%-complete" code populates public repositories, it becomes the training data for future models. This creates a negative feedback loop where models are increasingly trained on mediocre, context-poor, and potentially insecure code, threatening a long-term degradation in the quality of AI assistance across the industry.

#### The cognitive price of generation

The software industry has become obsessed with optimizing the easiest part of development - writing code - while largely ignoring the true bottleneck: _reading and understanding it_. AI code generation has effectively solved the "typing speed" problem, but in doing so, it has dangerously exacerbated the "comprehension speed" problem. The ease of generation creates a flood of code that developers must then painstakingly analyze to build the necessary mental model for true ownership.

True ownership of code is not defined by authorship. It is the possession of a robust and accurate _mental model_ of a system's behavior, its dependencies, and its failure modes. Without this internal map, a developer cannot debug effectively, anticipate the downstream impact of changes, or contribute confidently to the codebase. The code remains a "black box," an artifact whose behavior cannot be reasoned about, which is antithetical to the engineering discipline of building predictable, deterministic systems. The process of building this mental model is cognitively expensive, requiring a developer to trace logic across multiple files, understand complex data flows, and infer unstated intent from the text.

This dynamic inverts the traditional relationship between intent and artifact, fundamentally changing the nature of the engineering craft. Historically, a developer's mental model precedes the code; the code is an artifact of their thought process - a form of "forward motion". With AI assistance, this is reversed. The code artifact is generated first from a prompt, which is an imperfect and often ambiguous representation of intent. The developer must then become an "archaeologist," working backward from the generated artifact to reverse-engineer its logic and reconstruct the author's (in this case, the model's) intent. This process is fraught with challenges. AI code can be needlessly verbose and complex, turning what should be a 68-line function into a 620-line behemoth that is a nightmare to maintain. The AI can also confidently hallucinate and write broken code, requiring constant vigilance and deep scrutiny from the human reviewer. This creates a sense of detachment and erodes the feeling of craft, shifting the engineer's role from creator to validator.

This individual cognitive burden translates directly into a systemic organizational problem. The "dump-and-review" culture enabled by AI offloads the hard cognitive work of building a mental model from the author to the reviewer. The true cost of AI generation is not the price of the API call but the cumulative human cognitive hours required to safely integrate its output. This creates a massive bottleneck in the development process. A single developer can generate code far faster than a team can safely review and understand it, leading to the "AI productivity paradox" where individual velocity increases but team throughput stagnates or even declines. This dynamic also fuels trust debt and a diffusion of responsibility. The author can claim "the AI wrote it," while the reviewer is overwhelmed, leading to a situation where no single person truly owns the quality of the code. This is described by some as the worst form of tech debt imaginable.

This is precisely the problem this book will teach you how to handle. We will teach you the discipline for managing the erosion of the mental model and provide a framework that keeps the human engineer in cognitive control, not by rejecting AI, but by fundamentally reframing the work. The solution lies in making human-authored, executable specifications the non-negotiable source of truth and leveraging tooling and automation to ensure the AI's work rigorously conforms to our intent.

## 1.7 The beginning of “Software Engineering”

To conclude our introduction, we want to share one more thought that, like a lens, concentrates all the themes we’ve touched on. We are convinced that what is happening before our eyes is the inevitable end of the paradigm we long considered ideal: Software Craftsmanship. That movement, with its noble manifesto calling for "well-crafted software," taught us pride, responsibility, and the pursuit of mastery. Looking back, however, we see it was a trade-off for which we paid a high price. Today’s programming still resembles alchemy: full of meticulous recipes and artisanal apparatus. It's impressive, but in essence, it is a celebration of masterful yet manual practices, not the engineering practice.

The mistake we made as an industry was to uncritically adopt the metaphor of the craftsman. Craftsmanship, strictly understood, is handiwork: unique, dependent on a single master, and impossible to reproduce at the scale and consistency that engineering demands. This idealism led us to costly errors: "gold-plating," where we embellished code without real business value, and Sisyphean work, where each team reinvented the implementation of authentication, logging, or CI/CD from scratch. We paid for uniqueness with a lack of standardization, and for mastery with a lack of scalability of work. This was a fundamental, though often unnoticed, trade-off.

In this context, the arrival of AI agents capable of generating code at scale has become a brutal mirror, exposing our weaknesses. The discomfort many feel is not just an identity crisis, but proof of a deeper tension we have ignored for decades: we joked that software development isn't a true engineering discipline, failing to see that this joke described real errors in our approach. Too much tacit knowledge, too many rituals, and a lack of proven, universal methods. Alchemists had exquisite technique without a unifying theory; we’ve had beautiful code abstractions without consistent, executable specifications.

Paradoxically, it is precisely AI agents - perceived as a threat to "craft" - that are becoming the catalyst forcing us to take a painful but necessary step toward engineering. The paradigm we propose is not about blindly trusting the machine. It means consciously trading direct control over every line of code in exchange for the ability to design and validate the entire system that produces that code. It would be a mistake to think this is an easy exchange. We need to make the ultimate trade-off: we give up the satisfaction of manual creation and gain the immense burden of responsibility for a precise, ironclad, and verifiable specification that becomes the heart of the process.

This specification - a human-authored contract defining the system's behavior, performance, and security—is the missing link. It is the foundation of the repeatable and measurable production process that our discipline has lacked for so long. We are not witnessing the end of creative work, but its evolution, which brings its own pitfalls. Value no longer lies in tactical implementation prowess, but in the strategic ability to define intent.

This process is frustrating. What was once our "unspoken wisdom" and intuition must now become the language of measurable criteria: metrics, tests, and contracts. We lose the comfort of tacit knowledge and gain the burden of explicitly formulated requirements. This irritation, however, is the price we pay for the transition from craft to engineering.

This does not mean abandoning the spirit of the craftsmen. On the contrary, their skills become more crucial than ever. It is their "taste" and decision-making ability that guide the code generation process by AI. But here lurks the final, perhaps most important trade-off: we must codify this taste, transforming it from a personal preference into an explicitly defined, automatically verifiable set of rules.

That is precisely why we named this book _Vibe Engineering: Software Trade-Offs and Mistakes_. As an industry, we are currently reinventing a lot of techniques and must unlearn many things to move forward. In the following chapters, we will guide you through this new landscape, summarizing what we have learned so far - often the hard way. We will discuss specific challenges: how to migrate existing code in the AI era, how to precisely define context for generative agents, and how to collaborate with them effectively. We will also tackle tougher topics, such as the real costs of these technologies and the fundamental changes in managing engineering teams, as well as how AI-assisted engineering can help in staff-level tasks, like squeezing code performance metrics. The goal is not to avoid mistakes at all costs - the easiest way to do that would be to stick to the old-fashioned ways - but to make conscious decisions based on the knowledge gained from the ones we have already made. It is this journey through the world of difficult choices and unavoidable errors that we want to take you on

Wrapping up - it’s true, the industry is different and some well-taught routine has been taken from us to a degree, but in exchange we’ve received something greater. After years in the field, many of us hit a plateau: we recognize patterns faster and discover new ideas less often, the work becomes more re-creative than creative, and fresh abstractions no longer thrill as they once did. Not anymore. Today, it seems that since the 1960s, when the foundations of our field were laid, there hasn’t been a more fascinating moment for software engineering - a time when so much _new_ is being forged at once: new primitives, new workflows, new guardrails, new roles. With this book, we add our small contribution to that forging: presenting techniques turning the best of craft into a mature, repeatable method, which you will be discovering in following chapters.

- Extracting knowledge and understanding the bigger picture of an application can be done without diving into the codebase. Still, it requires us to avoid mistakes and existing habits of initially manually delving into the codebase.
- We developed a hybrid approach for the repository handling by combining the pros of the in-place and new repo approaches.
- The trade-off between too small and too big code increments done by an AI tool is an essential factor in the workflow. We tried to find a sweet spot between those two states where LLM suggests code increments that are not too small to make the process cumbersome, but also not too big to allow the human reviewer to understand the increment.
- Sometimes LLMs try to infer too much of the potential future needs, making the code over-engineered or even introducing bugs. We learn how to leverage this in some context, but also how to guard against this situation for unwanted cases.

# 4 Context Engineering: Optimizing context for AI Agents

### This chapter covers

- Why imprecise context leads to errors in AI agents
- The difference between prompt engineering and context engineering
- Strategies for writing, selecting, compressing, and isolating context
- Designing effective planning, execution, and debugging loops with AI agents
- Proven optimization techniques for achieving high code accuracy

An LLM doesn’t read minds - it feeds on the crumbs you throw at it.

You probably have heard about prompt engineering, which focuses on precisely formulating instructions for language models. Unfortunately, prompt engineering alone isn’t sufficient. It leads to limitations and so-called “context failures,” which are the primary cause of AI agent breakdowns - not errors of the model itself.

The solution to this problem is context engineering. This discipline provides the strategies for writing, selecting, compressing, and isolating context, as well as designing effective planning, execution, and debugging loops with AI agents. You will see how to apply these proven techniques for high code accuracy, using comprehensive examples with an easy-to-consume structure.

## 4.7 Summary

- Insufficient or imprecise context is a primary cause of AI coding tools failures, not errors of the model itself - as an insufficient or imprecise context causes human failures too. Isolating code snippets from the rest of the project leads to hallucinations and incorrect solutions.
- Context engineering is an evolution of prompt engineering; it focuses not just on formulating instructions but primarily on strategically selecting, structuring, and labeling information so the model can interpret it correctly.
- Manually collecting context is inefficient and prone to error. Automated tools using protocols like MCP allow agents to independently retrieve current and precise data from various sources like Figma, code repositories, and documentation systems.
- Providing more context does not always lead to better results. Language models suffer from the "lost in the middle" problem, meaning that critical information placed in the middle of a long prompt can be ignored, leading to a drop in accuracy. The same accuracy loss comes from overflowing the context window.
- AI coding tools automate context engineering with techniques like Retrieval-Augmented Generation to find relevant information, Context Anchors to maintain focus during long tasks, and Context Compaction to manage conversation history.
- AI tools do not eliminate the need for context engineering but instead shift the developer's role from a technician to a strategist. Rather than manually crafting prompts, the developer is responsible for curating the project's knowledge base, setting high-level direction, and scoping/correcting the tool's context-building process.
- To handle inconsistent or flawed context, agents must use advanced reasoning techniques. Chain-of-Thought compels the model to show its reasoning process , while Chain-of-Verification introduces a fact-checking loop, enabling a critical evaluation of the information it processes.
